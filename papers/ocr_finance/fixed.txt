                      摘 要

  随着企业日常报销业务逐渐增多，票据采集、字段录入、审批流转和资料留存
的工作量不断上升。传统人工登记方式需要反复录入发票代码、号码、日期和金额
等信息，容易出现错填、漏填和重复报销等问题，影响财务处理效率。针对这一实
际场景，本文设计并实现了一套基于 OCR 的智能财务报销系统，用于提高票据字
段采集效率并规范报销审批流程。
  基于 OCR 的智能财务报销系统采用 Spring Boot、Vue 和 MySQL 构建前后端分
离架构，主要实现员工报销申请、发票图片上传、OCR 识别、识别结果回填、重复
发票检测、财务审批、历史报销查询以及基础信息管理等功能。系统在识别环节接
入百度 OCR 接口，标准增值税发票优先调用 vatInvoice 接口获取结构化字段，异常
票据或字段不完整时再调用 basicAccurateGeneral 接口读取文本结果，并结合规则提
取金额、发票号码和开票日期等关键信息。
  测试结果表明，基于 OCR 的智能财务报销系统能够完成注册登录、报销填报 、
票据识别、重复检测、审批处理和信息管理等主要业务流程。以 20 张测试票据为
样本，清晰增值税发票的关键字段识别准确率达到 95%以上，重复发票能够被系统
拦截。系统运行结果说明，该设计能够满足中小企业常见财务报销场景的基本使用
需求。




关键词：智能财务报销系统 OCR 识别 重复发票检测 Spring Boot Vue




                         I
                                         Abstract

     With the increase of daily reimbursement work in enterprises, invoice collection, field
entry, approval circulation, and data archiving have become more frequent and time-
consuming. The traditional manual registration method requires repeated entry of invoice
code, invoice number, date, and amount, which may lead to input errors, omissions, and
repeated reimbursement. To solve these practical problems, this thesis designs and
implements an intelligent financial reimbursement system based on OCR recognition.
     The intelligent financial reimbursement system based on OCR is developed with Spring
Boot, Vue, and MySQL under a front-end and back-end separated architecture. The main
functions include employee reimbursement application, invoice image upload, OCR
recognition, automatic field filling, duplicate invoice detection, financial approval, historical
reimbursement query, and basic information management. In the recognition process, the
system integrates Baidu OCR services. The vatInvoice interface is used first for standard
VAT invoices, while the basicAccurateGeneral interface is used as a fallback when the
returned fields are incomplete. Key fields such as amount, invoice number, and invoice date
are then extracted through rules.
     The test results show that the system can complete registration and login, reimbursement
submission, invoice recognition, duplicate detection, approval processing, and information
management. In the test using 20 invoice samples, the key field recognition accuracy of
clear VAT invoices is higher than 95 percent, and duplicate invoices can be intercepted by
the system. The running results indicate that the proposed system can meet the basic needs
of common financial reimbursement scenarios in small and medium-sized enterprises.


Key words: Intelligent Financial Reimbursement System OCR Recognition Duplicate
Invoice Detection Spring Boot Vue

1.1 课题研究的背景
  在企业日常财务处理中，差旅、采购和接待等支出最终都要进入报销环节。申
请人提交票据后，财务部门还需要完成字段核验、审批和归档等工作。部分中小企
业仍以纸质票据和人工登记为主，金额、发票代码、发票号码和开票日期等字段需
要反复录入，报销单数量增加后，错填、漏填、审批滞后以及资料检索不便等问题
会更加明显。
  随着企业数字化建设持续推进，财务处理对效率和规范性的要求不断提高。
OCR 技术能够从票据图片中提取文字和结构化字段，为报销表单自动回填、重复发
票检测和后续审批提供数据基础。因此，将 OCR 识别能力接入财务报销流程，能
够减少手工录入环节，提高报销信息流转效率，也为中小企业财务管理的信息化改
造提供了现实条件。
1.2 课题研究的目的及意义
  本课题的研究目的，是设计并实现一套能够服务于中小企业报销场景的智能财
务报销系统。系统通过 OCR 识别发票图片中的关键字段，并将识别结果回填到报
销表单中，使员工在提交报销申请时减少重复录入；同时在提交前执行重复发票检
测，降低同一票据被多次报销的风险。通过这些功能，报销流程可以从纸质登记逐
步转向线上处理。
  从业务意义看，基于 OCR 的智能财务报销系统能够提高票据信息采集效率，
使财务人员在审批阶段直接查看识别字段、附件和报销说明，减少人工核验压力。
报销记录进入数据库后，还可以按照人员、时间和状态进行查询，为历史报销管理
和后续统计提供依据。
  从工程实践意义看，本课题涉及前后端分离开发、OCR 接口接入、数据库设计、
权限控制和业务流程联调等内容，能够较完整地体现软件工程专业项目开发过程。
通过该课题的设计与实现，可以进一步掌握管理类 Web 系统从需求分析到功能测试
的实现方法。

                     2 相关技术与工具
2.1 B/S 结构
   基于 OCR 的智能财务报销系统采用浏览器/服务器架构。用户侧通过浏览器访
问页面，业务处理、权限控制和数据读写集中在服务端完成，这种组织方式适合报
销审批这类以表单和列表操作为主的业务场景。
   与需要额外安装客户端的软件模式相比，B/S 结构更便于统一部署和后续维护。
系统升级主要发生在服务器端，员工和财务人员可以直接在浏览器中完成报销填报、
票据识别和审批处理。
2.2 Vue.js 框架
   前端界面基于 Vue 2 实现。该框架适合组织表单页、列表页和弹窗等常见管理
端页面，也便于把页面状态与数据模型直接关联起来，用于处理报销录入、审批切
换和结果展示。
   基于 OCR 的智能财务报销系统中，Vue Router 负责页面跳转，Vuex 保存登录
状态和部分全局数据，Axios 承担接口请求。组件化组织方式使报销表单、审批列
表和识别结果展示等功能块可以分别维护。
2.3 Element UI 组件库
   Element UI 为系统提供了较完整的管理端组件。表格、表单、上传控件、对话
框和分页器等部件基本覆盖了报销系统的大部分页面需求，也减少了基础交互代码
的重复编写。
   在具体实现中，报销记录列表由 el-table 展示，报销申请页通过 el-form 组织字
段，发票上传依赖 el-upload 完成文件选择与提交，审批确认则由对话框组件承载。
2.4 Java 语言
   后端程序使用 Java 编写。该语言在企业级 Web 开发中应用较广，相关框架和工
具链较为稳定，便于完成接口开发、权限控制、数据库访问和第三方服务集成。
   对本文而言，选择 Java 的主要原因在于其与 Spring Boot、MyBatis-plus 及百度
OCR Java SDK 的配合较为成熟，能够支撑报销业务所需的服务端实现。
2.5 Spring Boot 框架
   Spring Boot 承担基于 OCR 的智能财务报销系统的后端组织工作。它能够把控

制器、业务服务、数据访问和配置管理放在统一框架下处理，适合快速搭建面向业
务的 Web 服务。
   项目采用 Spring Boot 2.2.2 版本。其内嵌 Tomcat，部署方式相对直接；同时可
以较方便地整合权限拦截、统一返回结构和持久层组件，满足报销系统的接口开发

2.6 MyBatis-plus
   MyBatis-plus 主要用于简化持久层开发。常见的增删改查接口可以直接复用框
架提供的方法，从而减少基础 SQL 的手写工作量，让开发重点回到业务规则和条件
处理本身。
   系统在报销记录查询、员工信息维护和审批列表分页等场景中使用了条件构造
器与分页插件。依托这些能力，可以较方便地完成按状态、按用户或按时间区间的
检索。
2.7 MySQL
   系统数据库选用 MySQL，主要存放用户账号、报销单、识别字段和审核状态
等结构化记录。对于本课题这类中小规模 Web 系统，MySQL 在表结构组织、条件
查询和事务处理方面已经足够。
   报销提交、OCR 结果回填以及审批状态变更都通过关系表完成落库和关联，这
使得后续查询与统计能够围绕同一组业务数据展开。
2.8 百度 OCR API
   票据文字提取依赖百度 OCR 接口完成。该服务能够从图片中解析文本内容，
并针对部分票据类型直接返回结构化字段，适合接入报销录入流程。
   基于 OCR 的智能财务报销系统采用两类识别接口协同工作。标准增值税发票
优先使用 vatInvoice()接口，以获取发票代码、号码、日期和金额等字段；当专用接
口返回不稳定时，再切换到 basicAccurateGeneral()接口，对识别文本执行规则提取。
   在 工 程 实 现 中 ， 项 目 通 过 Maven 接 入 百 度 OCR 的 Java SDK aip-java-sdk
4.4.1。服务端创建 AipOcr 客户端后即可完成鉴权、图片提交和结果接收，底层请
求细节由 SDK 统一封装。

  服务端采用 Spring Boot，前端界面使用 Vue 与 Element UI，整体结构属于常见

3.2 功能性需求分析
  中小企业的报销业务通常围绕员工填报、票据核验和财务审核展开。若继续依
赖纸质单据与人工登记，字段录入、查重和状态追踪都会占用较多时间，流程也难
以沉淀为结构化数据。结合现有业务划分，基于 OCR 的智能财务报销系统保留管
理员、财务人员和员工三类角色。员工负责提交报销申请与查看处理进度，财务人
员负责审核与台账查询，管理员承担账号、角色和基础信息维护。
  员工是报销业务的发起者。其主要操作包括填写报销申请、上传发票图片、查
看识别结果、确认回填字段以及查询审批状态。票据上传完成后，系统调用 OCR
接口提取发票代码、号码、日期和金额等字段，员工核对无误后再提交申请。
  财务人员承担报销审核工作，并负责报销台账的查看与处理。其主要功能包括
查看待审批记录、核对票据与报销信息、执行通过或驳回操作，以及对历史报销数
据进行查询。OCR 识别结果会作为审核参考信息一并展示。
  系统管理员负责用户、角色和基础数据的维护工作。公告、留言、薪资等管理
功能属于系统运行中的辅助模块，用于支持日常信息维护，但不改变本文以 OCR
报销处理为主线的业务结构。

5.1 开发环境搭建
    系统开发前需要先完成运行环境配置。本项目的实验环境为 Windows 10，后端
使用 IntelliJ IDEA 2021，前端使用 Visual Studio Code，数据库管理工具为 Navicat
Premium。
    后端环境方面，首先安装 JDK 1.8 并配置环境变量，然后通过 Maven 管理项目
的 依 赖 包 。 项 目 引 入 spring-boot-starter-web 、 mybatis-plus-boot-starter 、 mysql-
connector-java、aip-java-sdk 等核心依赖，在 application.yml 中配置数据库连接信息、
服务端口号 8080 和上下文路径/qiyecaiwuguanli。
    前端环境方面，安装 Node.js 14.x 和 npm 包管理器，使用 Vue CLI 脚手架组织
Vue 2 项目。通过 npm 安装 element-ui、axios、vue-router、vuex 等依赖包，并在
vue.config.js 中配置开发服务器代理，使前端请求能够转发到后端接口。
    数据库部分使用 MySQL 8.0，数据库名称设置为 qiyecaiwuguanli，字符集为
utf8。完成库表导入后，再分别启动后端服务与前端开发服务器，浏览器即可访问
登录、报销、识别和审批相关页面。
5.2 员工功能模块的实现
    员工功能模块面向报销业务发起人开放，主要包括登录注册、报销信息提交、
发票图片上传、OCR 识别结果确认、薪资查询和留言管理等功能。其中，报销信息
提交与 OCR 识别是本课题实现的重点。
5.2.1 登录与注册
    用户进入基于 OCR 的智能财务报销系统后，首先到达登录页面。页面提供用
户名、密码和角色选择项，用户选择管理员、财务人员或员工角色后提交登录信息。
前端通过表单校验判断用户名和密码是否为空，校验通过后使用 Axios 向后端登录
接口发送请求。
    后端接收到登录请求后，会根据角色参数查询对应的数据表，并将用户输入的
密码与数据库中保存的加密密码进行比对。认证通过后，后端返回用户编号、角色
信息和令牌数据；前端将令牌写入 localStorage，并在后续业务请求中放入请求头。
若账号或密码错误，页面会给出提示并保持在登录页。登录界面如图 5-1 所示。

5.2.2 报销信息提交与 OCR 识别
    员工登录后进入报销信息管理页面，可以新增报销申请。报销表单包含报销类
型、报销金额、发票代码、发票号码、开票日期、报销说明和票据附件等字段。页
面通过 Element UI 的表单组件组织输入区域，并使用上传组件接收发票图片。
    当 员工上传发票图片后，前端先 把 文件提 交给 后端 OCR 识别接口。后端
OcrController 接收图片并转为 Base64 格式，再通过百度 aip-java-sdk 创建 AipOcr 客
户 端 对 象 。 系 统 优 先 调 用 vatInvoice() 方 法 识 别 标 准 增 值 税 发 票 ， 提 取
InvoiceCode 、 InvoiceNum 、 InvoiceDate 、 AmountInFiguers 、 PurchaserName 和
SellerName 等结构化字段。
    若 vatInvoice() 方 法 抛 出 异 常 或 返 回 字 段 不 完 整 ， 后 端 会 切 换 到
basicAccurateGeneral()方法进行通用文字高精度识别。通用接口返回文本行后，业
务层继续通过正则表达式提取金额、发票代码、发票号码和开票日期等字段，并统
一封装为前端可以展示的 JSON 结果。
    前端接收到识别结果后，将各字段展示在页面中供员工核对。员工确认无误后 ，
识别结果会自动回填到报销表单。提交报销申请时，后端会先依据发票号码查询报

销信息表，若发现重复记录则返回重复发票提示；若未重复，则将报销信息和 OCR
字段一起写入数据库，并把初始状态设置为待审批。报销信息填报与 OCR 识别界

5.3.1 报销审批

                     结 论

  本文围绕企业日常报销处理中票据录入重复、审批状态不易追踪和历史资料查
询不便等问题，完成了基于 OCR 的智能财务报销系统设计与实现。系统采用 Spring
Boot、Vue 和 MySQL 构建 B/S 架构，将员工报销申请、票据图片上传、OCR 识别、
字段回填、重复发票检测、财务审批和历史查询等环节组织到同一业务流程中，使
报销数据能够从提交阶段开始形成结构化记录。
  在系统实现方面，本文重点完成了三类角色的功能划分。员工端主要实现报销
申请、票据上传、识别结果确认和进度查询；财务人员端主要实现待审批记录查看、
票据核验、通过与驳回处理；管理员端主要实现用户、财务人员、薪资、公告和留
言等基础信息维护。OCR 识别模块采用增值税发票专用识别和通用文字识别相结合
的方式，标准票据优先返回结构化字段，异常样本再通过文本行结果进行规则提取，
从而增强系统对不同票据图片的适应能力。
  测试结果表明，基于 OCR 的智能财务报销系统能够完成预期功能。注册登录 、
报销提交、OCR 识别、重复发票检测、审批处理和信息管理等测试用例均能得到与
预期一致的结果；在 20 张样本票据测试中，清晰增值税发票关键字段识别准确率
达到 95%以上，重复发票能够被系统正确拦截。兼容性和性能测试结果也说明，系
统在常见浏览器环境下可以稳定运行，页面和接口响应时间能够满足毕业设计演示
与中小企业基础报销场景的使用要求。
  由于开发时间和测试样本有限，系统仍存在可改进之处。一方面，票据图片模
糊、角度偏斜或票据类型差异较大时，识别结果仍可能需要人工修正；另一方面，
当前审批流程主要按单级审核设计，对多级会签、预算控制和移动端审批等复杂业
务支持不足。后续可从图像预处理、识别策略扩展、审批规则配置和移动端适配等
方向继续完善，使系统在真实财务管理场景中具备更强的适用性。
