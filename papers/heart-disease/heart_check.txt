1 概述
1.1 研究背景
心血管疾病杀死的人比任何一种病都多——这个排名已经持续了很多年。WHO

在 2023 年给出的数字是：每年大约 1790 万人死于心血管疾病，占全球死亡总数的

32%上下。国内的情况也差不多，不管城市还是农村，心血管疾病都是排第一的死

因。《中国心血管健康与疾病报告 2022》里面的估算是：全国心血管病患者大约

3.3 亿，光冠心病就超过 1100 万。心脏病不是单一原因造成的。年龄、性别、吸不

吸烟、有没有糖尿病和高血压这些因素会一起起作用，而且因素之间还会互相影响 。

要是能在发病之前把高风险的人找出来，让他们改改生活习惯或者提前吃药，发病

率和死亡率是可以明显降下来的。
传统的心脏病风险评估主要依赖医生的临床经验和单一指标的阈值判断。

Framingham 风险评分是目前应用最广的工具之一，它用年龄、总胆固醇、 HDL 胆

固醇、收缩压、吸烟状态等几个变量算出未来 10 年的冠心病风险概率。这种方法

简单直观，但局限也很明显——变量数量有限，无法纳入 BMI、睡眠时长、精神健

康天数等维度的信息；对不同人群的适用性也受到质疑。

1

大数据技术这几年发展得比较快，这给心脏病风险分析打开了一条新路。网上

能找到好几个大规模的健康调查数据集，里面记录了几十万人的健康行为和疾病情

况。Hadoop 和 Hive 这类分布式框架拿来处理这些数据正好合适——清洗、转换、

多维聚合都能跑得动。聚合完的数据拿去喂机器学习模型，模型可以同时看多个维

度的风险因素，预测粒度比传统的评分卡细得多。分析结果再用 ECharts 之类的可

视化工具画出来，医护人员和普通人看着都比较直观。

1.2 国内外研究现状
心脏病风险预测是机器学习在医疗领域最早的应用方向之一。 UCI Machine

Learning Repository 在 1988 年就公开了 Cleveland 心脏病数据集，包含 303 条记录

和 13 个临床指标。Robert Detrano 等人在该数据集上用逻辑回归做二分类，准确率

达到 77%。此后大量研究以此为基线展开改进。
后面陆续有人用集成学习和深度学习来做，准确率确实上去了不少。比如

Mohan 等 人 （ 2019 ） 把 随 机 森 林 和 线 性 模 型 混 在 一 起 搞 了 个 HRFLM 方 法 ，

Cleveland 数据集上跑出 88.7%的准确率。Ali 等人（2021）更狠，一口气跑了 14 种

分类器，结果是 Extra Trees 和随机森林表现最好，AUC 都超过了 0.91。Kaggle 平

2

台发布的 Indicators of Heart Disease 数据集（基于 CDC BRFSS 2020，319795 条

记录、18 个字段）也带火了一波分析，数据量比传统的临床数据集大了好几个数

量级。
Hadoop 和 Hive 用在医疗健康方向的案例也不少。Viceconti 等人（2015）讨论

过 Hadoop 生态在临床数据仓库里怎么落地。国内这边，朱文球等人（ 2020）用

Hive+Spark 搭了慢性病数据分析平台，从数据采集到可视化的链路跑通了。曹丽娜

等人（2022）的工作跟本文比较接近——也是心脏病方向，MapReduce 清洗数据，

Hive 建仓库，Pyecharts 画大屏。
综合来看，现有研究存在两方面不足。一是大多数机器学习研究只用一个小规

模临床数据集，缺少对多数据源、大规模调查数据的整合分析。二是不少大数据平

台类的工作止步于可视化展示，没有把机器学习预测和在线服务做进去。本文试图

在这两个方面做改进：既在 Hadoop 数据仓库中整合多个数据集做分析，又在此基

础上训练预测模型并提供 Web 在线预测服务。

1.3 研究目的
这个课题要做的事情就是搭一套基于 Hadoop 的心脏病数据分析和预测系统。

3

功能上要跑通四个环节：离线数据仓库、多维统计分析、机器学习预测、Web 可视

化展示。
为什么要做这个？数据层面，BRFSS 调查数据和 UCI 临床数据的心脏病数据集

格式不一样、字段不一样，放到 Hadoop 上用 Hive 统一建 ODS-DWD-ADS 三层仓

库，后面不管做分析还是建模都有一个干净的数据底座。模型层面，同一份数据跑

多种算法然后横向对比，哪个模型好用不是拍脑袋说的，是数字说的。应用层面，

分析结果和预测能力如果只能在命令行里跑，除了开发者谁也用不了——封装成

Web 系统，打开浏览器就能看分析、填指标、拿风险评估。

1.4 项目规划
本系统基于 Hadoop 大数据平台和机器学习技术，构建心脏病健康数据的完整

处理链路。具体规划如下六个模块：
1. 多源数据整合：整合 Indicators of Heart Disease（2020）数据集（319795 条

18 字段）、Indicators of Heart Disease（2022 UPDATE）数据集（246022 条、40

字段）和 UCI Cleveland 临床数据集（303 条、14 个临床指标），覆盖健康行为调

查和临床检测两类数据源。
2. 数据仓库构建：在 Hadoop 平台上通过 Hive 构建 ODS-DWD-ADS 三层数据
4

仓库。ODS 层原样存储 CSV 数据，DWD 层完成类型转换、字段标准化和缺失值处

理，ADS 层按年龄、性别、BMI、生活习惯、共病史和 UCI 临床指标六个维度做

聚合统计。
3. 多维统计分析：基于 ADS 聚合表从六个维度展示心脏病患病率的分布特征，

包括年龄段的指数增长趋势、性别差异、生活习惯影响、共病关联强度和睡眠时长

的 U 形分布等。
4. 机 器 学 习 预 测 ： 在 Indicators of Heart Disease （ 2020 ） 数 据 集 和 UCI

Cleveland 两个数据集上训练逻辑回归、SGD 线性分类器、随机森林和 Extra Trees

四种模型，以 AUC 为主要选型标准进行对比评估，选出最优模型用于在线预测。
5. 数据可视化展示：前端采用 Vue3 配合 ECharts 搭建可视化大屏和多页分析界

面，展示各维度的统计图表、模型评估指标和特征重要性排名。
6. 在线预测服务：后端基于 Django REST Framework 提供数据查询和在线预

测 API 接口，用户输入健康指标后即可获得心脏病风险等级和关键影响因素。整套

系统用 Docker Compose 容器化部署，前后端分离。

2.1.3 机器学习预测功能
机器学习模块提供模型训练结果的查看和在线预测两项功能：
模型对比：展示逻辑回归、SGD 线性分类器、随机森林和 Extra Trees 四种模

型在 Indicators of Heart Disease（2020）数据集和 UCI Cleveland 两个数据集上的

Accuracy、Precision、Recall、F1 和 AUC 五项指标，以分组柱状图、雷达图和 ROC

曲线呈现。
特征重要性：展示影响心脏病风险的 Top-10 特征排名及其重要性数值，支持按

数据集切换查看。
在线预测：提供一个包含 16 个健康指标字段的输入表单（BMI、年龄段、性别、

吸烟、饮酒、运动习惯、整体健康状况、睡眠时长、行走困难、中风、糖尿病、哮

喘、肾病、皮肤癌、身体健康天数、精神健康天数），用户填写后提交即可获得风

险概率、风险等级（低/中/高）和影响当前结果最大的三个因素。缺失字段由模型

训练时记录的默认值自动补全。
在线预测功能要求训练阶段和推理阶段使用完全一致的特征处理流程。系统不

能在前端直接计算风险概率，而应由后端加载训练好的 Pipeline 对象完成预处理和

模型推理，避免不同端实现不一致导致结果偏差。

3.1 技术支撑
3.1.1 大数据端
大数据端负责原始数据的存储、清洗、分析和迁移四个环节：
（1）数据存储。三个 CSV 数据文件（Indicators of Heart Disease（2020）、

Indicators of Heart Disease（2022 UPDATE）、UCI Cleveland）存储在 HDFS 上。

Hadoop 集群采用 Docker 方式搭建，包含 1 个 NameNode 和 2 个 DataNode，HDFS

副本数设为 2。HDFS 把文件切成固定大小的块（默认 128MB），分散放到集群里

不同的机器上，每个块存 3 份。NameNode 记的是元数据——哪个文件切成了哪几

个块、块在哪台机器上。DataNode 就管读写数据本身。
（2）数据清洗。Hive 是建立在 Hadoop 之上的数据仓库工具，用户写 SQL

（HiveQL），Hive 把 SQL 翻译成 MapReduce 或 Tez 任务在集群上执行。本系统在

Hive 上构建 ODS-DWD-ADS 三层数据仓库。ODS 层通过外部表直接映射 HDFS 上

的 CSV 文件，字段全部以 STRING 类型载入；DWD 层通过 INSERT OVERWRITE

将 STRING 转换为 DECIMAL、INT 等精确类型，同时做字段标准化和缺失值处理；

16

ADS 层用 GROUP BY 和窗口函数做多维聚合。
（3）数据分析。ADS 层产出 9 张聚合表，覆盖年龄、性别、BMI、生活习惯、

共病史和 UCI 临床指标六个分析维度。每张表的粒度为数据集×维度值的交叉组合，

输出样本数、阳性数和患病率三个核心指标。Hive 的分区和分桶机制可以加速查询

——本系统按 dataset_name 做分区，查询某个数据集时只需要扫描对应分区的文件。
（4）数据迁移。ADS 层的聚合结果通过 Python 脚本（PyHive + PyMySQL）

同步到 MySQL。同步方式为全量覆盖：先 TRUNCATE 目标表再批量 INSERT。

ADS 表每张只有几十到几百行，全量同步耗时在秒级。MySQL 同时存储用户账户

信息和预测历史记录。

3.1.2 机器学习预测
预测模块使用 scikit-learn 训练四种分类模型：
逻辑回归是最基础的二分类模型。它对输入特征做线性加权求和后过 Sigmoid

函数，输出 0 到 1 之间的概率值。训练快、可解释性强，缺点是只能学到线性决策

边界。设置 class_weight='balanced'来应对正负样本不均衡。
随机森林由 250 棵决策树组成。每棵树在训练时只看到一部分样本和一部分特

征，最终预测结果由所有树投票决定。随机性降低了过拟合风险，集成多棵弱分类

器也提高了泛化能力。
Extra Trees（Extremely Randomized Trees）和随机森林的区别在于：随机森林

19

在每个节点选最优分裂点，Extra Trees 随机选分裂点。这让 Extra Trees 训练更快，

方差更低，但偏差略高。本系统配置了 300 棵树。
SGD 线性分类器用随机梯度下降来训练逻辑回归，适合大规模数据集。loss 设

为 log_loss 时等效于逻辑回归，但训练速度更快。
四种模型的选择以 AUC（ROC 曲线下面积）为首要标准。AUC 衡量的是模型

区分阳性和阴性样本的总体能力，不受分类阈值的影响，在正负样本不均衡的情况

下比准确率更有参考价值。

4.1.3 DWD 层实现
DWD 层接 ODS 层的原始数据，主要做三件事：清洗脏数据、转数据类型、统

一字段名。处理完之后就是能直接拿来分析的明细数据了。
清 洗 过 程 通 过 HiveQL 的 INSERT OVERWRITE 语 句 实 现 。 以 Indicators of

Heart Disease（2020）数据为例，主要的清洗操作包括：（1）BMI 从 STRING 转

38

为 DECIMAL(6,2) ； （ 2 ） Smoking 、 AlcoholDrinking 等 Yes/No 字 段 通 过 CASE

WHEN 映 射 为 1/0 的 INT 类 型 ； （ 3 ） Sex 字 段 从 Male/Female 映 射 为 1/0 的

sex_code ； （ 4 ） HeartDisease 字 段 映 射 为 risk_label （ Yes→ 1 ， No→ 0 ） ； （ 5 ）

Diabetic 字段保留原始文本值（diabetic_text）同时生成二值标记（diabetic_flag）。
UCI Cleveland 数据集的问题主要是缺失值。ca 和 thal 两个字段里有问号（?），

用 CASE WHEN 换成了 NULL。目标变量 num 原来是 0 到 4 五个值，这里简化成

二 分 类 ： num>0 就 算 有 病 ， risk_label 设 为 1 。 年 龄 按 10 岁 一 档 分 段 ， 方 便 跟

BRFSS 调查数据对齐。
DWD 层最关键的一张表是 dwd_heart_feature_sample。做法是把三个数据集都

有的维度（risk_label、age_band、sex_code、bmi、smoking_flag 这些）用 UNION

ALL 拼成一张 28 列的宽表。BRFSS 调查数据中没有的临床指标（trestbps、chol 之

类）填 NULL，UCI 那边没有的生活习惯字段也填 NULL。每条记录多挂一个

source_dataset 字段标来源，ADS 层按数据集分组的时候要用。
DWD 清洗脚本按字段类型分为三类处理。第一类是数 值字段，如 BMI、

PhysicalHealth、SleepTime，通过 CAST 转换为 DECIMAL 或 INT；第二类是二值字

39

段 ， 如 Smoking 、 Stroke 、 KidneyDisease ， 通 过 CASE WHEN 把 Yes/No 映 射 为

1/0；第三类是分组字段，如 AgeCategory 和 BMI 分组，用于后续统计展示。
在统一特征表 dwd_heart_feature_sample 中，BRFSS 调查数据和 UCI 临床数据

的字段并不完全相同。为保证 UNION ALL 能够执行，系统先定义统一字段清单，

再让各数据源按同样顺序输出字段。某个数据源没有的字段统一填 NULL，例如

BRFSS 调查数据没有 trestbps、chol、thalach 等临床指标， UCI 没有 Smoking、

AlcoholDrinking 等生活习惯指标。
清洗完成后需要进行三类校验：第一，COUNT(*)核对清洗前后记录数，确认

没有误删；第二，检查 risk_label 只能取 0 或 1；第三，抽样查看 BMI、SleepTime

等数值字段是否存在异常范围。只有校验通过后，DWD 表才进入 ADS 聚合流程。
4.1.4 ADS 层实现与数据迁移
ADS 层的 SQL 统一使用 GROUP BY 做分组聚合。以 ads_heart_by_age 为例，

按 dataset_name 和 age_band 两 个 维 度 分 组 ， COUNT(*) 统 计 样 本 数 ，

SUM(risk_label) 统 计 阳 性 数 ， SUM(risk_label)/COUNT(*) 计 算 患 病 率 。

ads_heart_lifestyle 将吸烟、饮酒、运动、睡眠四个因素用 UNION ALL 拼成统一的

factor_name/factor_value 格 式 ， 每 行 输 出 一 个 因 素 值 的 统 计 结 果 。

的每个取值各输出一行。
ADS 层的聚合结果通过 Python 同步脚本迁移到 MySQL。脚本流程为：PyHive

4.2 预测模型设计
4.2.1 特征工程
特 征 工 程 基 于 DWD 层 的 dwd_heart_2020_clean 表 。 数 值 型 字 段 （ BMI 、

PhysicalHealth、MentalHealth、SleepTime）做 StandardScaler 标准化，将均值调整

为 0 、 标 准 差 调 整 为 1 。 分 类 型 字 段 （ Smoking 、 AlcoholDrinking 、 Stroke 、

DiffWalking、Sex、AgeCategory、Race、Diabetic、PhysicalActivity、GenHealth、

Asthma、KidneyDisease、SkinCancer 共 13 个）做 OneHotEncoder 独热编码，将每

个类别值展开为一个二值列。
scikit-learn 的 ColumnTransformer 将两路处理组合为 Pipeline，保证训练和推理

逻辑一致。数据按 8:2 划分训练集和验证集，分层抽样保持正负样本比例一致，随

机种子 42。

特征工程不仅影响模型精度，也影响线上预测接口的稳定性。训练阶段使用的

类 别 不 均 衡 是 心 脏 病 预 测 数 据 中 的 主 要 问 题 。 Indicators of Heart Disease

数样本判为阴性。本文在逻辑回归和 SGD 模型中设置 class_weight='balanced'，让

阳性样本在损失函数中获得更高权重，从而提升召回率。

4.5 前端框架设计
前端基于 Vue3 Composition API 构建，主要技术组件包括：
Vue Router 负责页面路由管理。配置了路由守卫（beforeEach），未登录用户

自动跳转登录页；管理员页面通过 meta.requiresAdmin 标记，普通用户访问时重定

向到首页。路由采用懒加载（dynamic import），减少首屏资源体积。
Pinia 替代 Vuex 作为全局状态管理方案。定义了 useUserStore（用户登录状态、

角色信息）和 useAnalysisStore（缓存分析数据，避免重复 API 请求）两个 Store。

分析页面切换时，如果 Store 中已有对应维度的数据则直接渲染，不再重复请求后

端。
ECharts 集成方面，封装了 useChart 组合式函数，统一管理图表实例的创建、更
55

新和销毁。图表组件通过 props 接收数据和配置，内部使用 watch 监听数据变化自

动重绘。柱状图、折线图、饼图、雷达图和 ROC 曲线图各封装为独立组件。
Axios 封装方面，创建 axios 实例并配置 baseURL、超时时间和拦截器。请求拦

截器从 Pinia 的 userStore 中读取 JWT 令牌附加到 Authorization 头；响应拦截器统一

处理 401（令牌过期）、403（权限不足）和 500（服务器错误）。
CSS 布局方面，大屏模式使用 CSS Grid 定义多个网格区域，每个图表占据指

定的 grid-area。分页管理模式使用 Naive UI 的 NLayout + NMenu 组件构建侧边栏

导航。
前端页面按业务域拆分为五组路由：dashboard 用于可视化大屏，analysis 用于

年龄、生活习惯、临床指标和综合对比， model 用于模型指标和特征重要性，

predict 用于在线预测，admin 用于用户、日志和同步管理。每个页面只负责渲染自

身业务数据，公共请求逻辑放在 api 目录中统一封装。
预测页面的交互流程为：用户填写表单后点击开始预测，页面先执行前端基础

校验，再调用/api/predict 接口。接口返回后，页面右侧展示风险概率、风险等级和

影响因素 Top3，下方预测历史表格自动刷新。若接口返回 400 错误，表单项附近显

56

示具体错误；若返回 401，路由守卫跳转到登录页。
为了提升图表页面体验，ECharts 组件在 mounted 阶段初始化实例，在 watch 中

监听数据变化，在 beforeUnmount 中销毁实例。窗口大小变化时触发 resize，保证

大屏模式和分页模式都能正常自适应。

结 论

这篇论文围绕心脏病健康数据，做了四件事：建数据仓库、跑多维统计、训练

机器学习模型、开发 Web 系统。数据这块，三份公开数据集走了 ODS-DWD-ADS

三层 Hive 仓库，清洗和标准化之后产出 9 张 ADS 聚合表，分别对应年龄、性别、

BMI、生活习惯、共病史、UCI 临床指标六个维度。模型这块，Indicators of Heart

Disease（2020）数据集上逻辑回归 AUC=0.8207、召回率 78.30%，排第一。UCI 数

据上同一模型 AUC 到了 0.9665。系统这块，Vue3+ECharts 负责可视化大屏和分页

分析界面，Django 后端负责 RESTful 查询和在线预测，Docker Compose 负责容器

化部署。

目前的 问题 也比较明显 。 Indicators of Heart Disease （2020）数据集 阳性只有

8.6%，正负样本差太多，模型精确率只有 23.31%。数据仓库只能离线批处理，没

接实时数据流。预测模型也只输出一个概率值，没有可解释性分析。

后面如果要改，样本不平衡的问题可以试 SMOTE 过采样或者 Focal Loss；实时性

69

的问题可以在 Hive 旁边加一路 Spark Streaming；可解释性的问题可以用 SHAP 值

给每个预测结果标出哪些特征影响最大。
